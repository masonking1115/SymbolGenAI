 HW/SW CO-DESIGNER  —  PIPELINE & WORKFLOW OVERVIEW                test1/
===============================================================================
 Plain-text companion to docs/pipeline.tex. The .tex covers the schematic-
 GENERATION build in depth; this file covers the whole system end to end, with
 emphasis on the parts the .tex doesn't: the SIMULATION subsystem and the AI-
 AGENT workflows that drive the GUI. Keep both in sync when the pipeline changes.

 Three pipelines share one set of inputs (the YAML netlists + per-MPN symbols):

   A. GENERATION pipeline — deterministic Python compiler: YAML + symbols
      -> validated Altium schematic project (.SchDoc/.PrjPcb/.svg).  No AI.
   B. SIMULATION pipeline — behavioral SPICE models of testable sub-circuits,
      run via ngspice, with AI agents for datasheet extraction, result
      interpretation, an iteration loop, and SPICE-model lifecycle
      (generate / update-to-match-schematic / chat-edit).
   C. DESIGN-REVIEW pipeline — a rule set (113 rules) evaluated structural +
      semantic (claude-judged) + sim_review, driving a closed loop that FINDS
      and then FIXES issues, ending in Diff & Accept.

 Everything runs through the Altium spike venv (has altium_monkey, PyMuPDF,
 fastapi, ngspice on PATH):
   PY = c:/Users/mking/Downloads/altium_spike/.venv/Scripts/python.exe

 The netlist is the single source of truth for all three pipelines. The
 generation pipeline renders it to a schematic; the simulation pipeline reads
 as-built component values from it (via sim/design_extract.py) so a design change
 flows into the sims with no hand-editing; the design-review pipeline evaluates
 the netlist + built schematic + datasheets and can edit the netlist/builders to
 FIX what it finds.


===============================================================================
 A.  GENERATION PIPELINE  (deterministic — summary; see docs/pipeline.tex)
===============================================================================

 A "deterministic compiler" for schematics: design intent expressed once in
 YAML, rendered to Altium-native files by pure-Python builders. No manual
 drawing; every wire/pin/label/power-symbol is emitted programmatically and
 every build is gated.

 Entry point:   PY -m test1.altium.build_project
 Backend lib:   altium_monkey (PyPI, pinned v2026.5.26) — reads/writes native
                Altium binaries with NO Altium install; ships an in-process SVG
                renderer (to_svg) used as a "does it parse+render?" gate.

 Stages:
   1. Symbol library build  (altium/build_symbols.py)
        merge every Parts Library/<MPN>/<MPN>.SchLib (+ auto-authored passives)
        -> altium/out/lib/parts.SchLib.  Standard body glyphs via altium/glyphs.py
        (R zig-zag, cap plates, MOSFET w/ body diode, op-amp triangle); pins
        (hot-spots) untouched so routing/validator/linter are unaffected.
   2. Sheet building  (altium/shared.py : AltiumSheet)
        per-sheet builders (build_power, build_bias, build_bobcat, build_fmc,
        build_connectors, build_eeprom) place_from_netlist + wire/junction/
        port/power_at/text.  Accumulates duck-typed connectivity records
        (_placed/_wires/_junctions/_labels) both gates walk without re-parsing.
   3. Connectivity validation  (gen/validator.py)  — HARD GATE, fails the build
        union-find over pins + wires (T-intersections auto-connect) + labels;
        each YAML net must resolve to one connected component.
   4. Layout lint  (altium/layout_lint.py)  — QUALITY GATE
        text-as-2D-boxes geometry checks; RULES registry (35 rules: 6 ERROR /
        26 WARNING / 3 INFO) is the single source shared by checks + build report
        + GUI checklist.  ERROR fails the build; WARN/INFO advisory.  (Full rule
        table in pipeline.tex.)
   5. Centering + project assembly  (altium/build_project.py)
        build each sheet twice (measure bbox, then grid-snapped centered shift),
        write *.SchDoc + root + test1.PrjPcb, print per-sheet E/W/I summary.
   6. GUI consumption  — /api/sheets, /api/png/<sheet>, /api/lint.

 Clean run: per sheet E/W/I = 0/0/0 (cosmetic redundant_junction INFO expected),
 "FAILURES: none".

 Sheets:  power · bias · bobcat · fmc · connectors · eeprom  (+ root)


===============================================================================
 B.  SIMULATION PIPELINE  (sim/  +  ngspice  +  AI agents)
===============================================================================

 The sim layer verifies architecture + component values by simulating testable
 sub-circuits ("blocks"). The decks are HAND-WRITTEN BEHAVIORAL SPICE models
 (regulation, gain, droop) — NOT a transistor-level transcription of the
 schematic — but their component VALUES are read from the as-built netlist, so
 the sim tracks the design.

-------------------------------------------------------------------------------
 B.1  The block catalog  (sim/blocks.yaml  via sim/catalog.py)
-------------------------------------------------------------------------------
 Each entry = one testable sub-circuit: which netlist sheet it comes from, which
 sim_types apply (each with a rationale + a concrete `pass` criterion), the
 device models it needs, the off-sheet boundary stubs, and a functional `group`.

 BOUNDARY vs EXTRACTED (the key split):
   • blocks.yaml holds ONLY genuinely off-sheet boundary values — source
     impedance (R_src/L_src = layout), enable timing (t_on = FPGA firmware),
     downstream load (I_base/I_step = DUT behavior), incoming rail V, and the
     `pass:` acceptance criterion.  These are curated estimates not in the
     schematic.
   • COMPONENT VALUES that ARE in the design (decoupling caps, the bias sense
     resistor, jumpers) are NOT in blocks.yaml — the decks read them from
     netlist/<sheet>.yaml via sim/design_extract.py.  A design change flows
     straight into the sim.

 Functional groups (GUI sidebar sections; ordered/labeled by SIM_GROUPS in
 app.py, tagged per block by `group:`):
     power_generation     — how a rail is made (LDO + load switch)
     power_distribution   — how a rail is delivered (decap/PDN banks)
     analog_bias          — precision analog (the V-to-I bias loop)
     system_integration   — multi-block / whole-board tests
     not_simulatable      — documented gaps (connectors, FMC, EEPROM)

 Block status:  implemented (deck runs) · planned (curated, no deck yet) ·
                not_simulatable (no useful SPICE behavior, documented).

 Current blocks (6 implemented blocks → 14 implemented sims, all 14 PASS):
   ldo_rail        power_generation     implemented   (TPS7A8401A + TPS22916)
       dc_op_point · transient_powerup · transient_load_step · line_regulation
       · setpoint_coverage   (5)
   opa_bias        analog_bias          implemented   (OPA2388 V-to-I loop)
       dc_sweep · dc_compliance · ac_stability · transient_settling · por_failsafe (5)
   vddio_pdn       power_distribution   implemented   (VDDIO decap bank)   transient_load_step
   vddd_pdn        power_distribution   implemented   (VDDD decap bank)    transient_load_step
   vdda1_pdn       power_distribution   implemented   (VDDA1 decap)        transient_load_step
   vdda2_pdn       power_distribution   implemented   (VDDA2 decap)        transient_load_step
   system_sequencing  system_integration  planned     (whole-board bring-up)
   eeprom / fmc / connectors            not_simulatable

-------------------------------------------------------------------------------
 B.2  Deck builders  (sim/decks/*.py)  — the DECK-BUILDER CONTRACT
-------------------------------------------------------------------------------
 One module per block: ldo_rail.py, opa_bias.py, pdn.py.  Each exposes
   build_deck(*, mode, **kw) -> (ngspice deck text, trace_specs)
 plus analyzer fn(s) returning {... "overall": "OK"|"FAIL"} and a refdes_map()
 tying each deck element ref to its netlist refdes (or None for behavioral
 scaffolding: boundary sources, ammeters, DUT load stubs).

 A conformant deck:
   1. build_deck(*, mode, **kw) -> (deck_text, trace_specs).
   2. Reads as-built values from sim/design_extract.py (caps_on_net,
      resistor_value, sense_resistance, ...). NEVER hardcodes a netlist value.
   3. Uses shared device models (sim/models.py) + boundary stubs (sim/stubs.py
      emit_stub) rather than inline reinvention.
   4. Provides analyzer(s) with an "overall" OK/FAIL key + a refdes_map().
   5. Is wired into sim/service.py dispatch (run_block_sim + build_deck_text +
      _refdes_map_for) AND has a blocks.yaml entry.
   6. Stamps provenance once it matches the schematic
      (python sim/deck_provenance.py --stamp <block_id>).
   7. Verifies: service.run_block_sim(block, sim_type)['status'] == 'ran'
      (or 'no_simulator' if ngspice is absent — still means it built+dispatched).

-------------------------------------------------------------------------------
 B.3  sim/ module map
-------------------------------------------------------------------------------
   blocks.yaml          curated block catalog (the library surfaced in the tab)
   catalog.py           load catalog; resolve_boundaries (base + per-sim merge)
   catalog_edit.py      surgical, comment-preserving blocks.yaml edits
   service.py           GUI-facing dispatch: block+sim_type -> build deck -> run
                        ngspice -> analyze -> JSON (traces, analysis, circuit).
                        has_deck_builder() = does a block have a SPICE model.
   runner.py            invoke ngspice -b on a deck; parse .dat traces
   models.py            shared behavioral device models (LDO, switch, op-amp, FET)
   stubs.py             boundary stubs (RailIn/RailOut/DigitalDrive/...) emit_stub
   design_extract.py    THE reader of netlist values for the sim layer
                        (parse_value w/ tolerance strip, caps_on_net,
                        resistor_value, sense_resistance, sheet_refdes)
   simconfig.py         operating-point SCENARIO cache + freshness (is_fresh)
   dscache.py / param_map.py  datasheet-derived device params -> ngspice model
                        inputs (param_map._MAP keys the parts that feed models)
   circuit.py           parse a deck -> {elements, nets, subckts} node-graph for
                        the GUI "SPICE model" view (refdes-aware)
   deck_provenance.py   per-block netlist FINGERPRINT + staleness:
                        has_model / model_status = none|unknown|fresh|stale
   iterate_sim.py       bounded re-sim CLI for the interpret agent (HARD cap 3)
   read_pdf.py          render datasheet PDF pages to PNG (PyMuPDF) +
                        --text-only fast path, so agents can read datasheets
                        (the Read tool can't rasterize PDFs here — no poppler)
   fault_injection.py / run_sim.py   supporting helpers

-------------------------------------------------------------------------------
 B.4  THE SIMULATION RUN WORKFLOW  (what "Run" does in the Simulation tab)
-------------------------------------------------------------------------------
 3 stages, context-first (params decided BEFORE the sim runs), with an outer
 feedback loop. Each AI stage is a `claude -p` subprocess streamed over SSE.

   ┌──────────────────────────────────────────────────────────────────┐
   │ STAGE 1  SETUP   (AI: "Datasheet & scenario agent")  cache-gated   │
   │   POST /api/sim/setup -> start_sim_setup                           │
   │   If the block's scenario+params are FRESH (simconfig.is_fresh),   │
   │   SKIP. Else the agent reads datasheets (read_pdf, TEXT-FIRST) +   │
   │   requirements + the netlist, then writes:                        │
   │     - device params  -> sim/cache/datasheet_params.json           │
   │     - operating point-> sim/cache/sim_config.json                 │
   │   Setup OWNS datasheet extraction; interpret only consumes it.    │
   ├──────────────────────────────────────────────────────────────────┤
   │ STAGE 2  SIMULATE   (ngspice — no AI)                              │
   │   POST /api/sim/run -> service.run_block_sim                       │
   │   build deck (values from netlist) -> ngspice -b -> analyze        │
   │   -> {ok, status, analysis, plot traces, op_point, deck, circuit}  │
   ├──────────────────────────────────────────────────────────────────┤
   │ STAGE 3  INTERPRET  (AI: "Verdict agent")                         │
   │   POST /api/sim/interpret -> start_sim_interpret                   │
   │   cross-checks the result vs datasheet specs + requirements;       │
   │   emits VERDICT (MEETS_SPEC | OUT_OF_SPEC | NEEDS_CLARIFICATION),  │
   │   MARGIN (with a datasheet citation), SUGGESTIONS (circuit edits   │
   │   -> changelog), CLARIFY, ITERATIONS.                              │
   │   INLINE ITERATE: may fix a wrong SCENARIO param (sim_config) and  │
   │   re-run via sim/iterate_sim.py — HARD-capped at 3 re-sims.        │
   └──────────────────────────────────────────────────────────────────┘

 OUTER FEEDBACK LOOP (workflow visual, up to 3 passes): if a pass comes up
 incomplete, the reviewer's feedback informs a re-tune and the setup->sim->
 interpret sequence re-runs. The tab's workflow strip shows a green check per
 completed step and a "loop N/3" badge once it passes the first pass.

 RESILIENCE: the interpret SSE has a watchdog + settles-on-error so a dropped
 stream / backend restart can't hang the Run; a Cancel button terminates the
 live claude -p; persisted state never shows phantom "running" on reload. All
 sim-cache I/O (sim_config.json / datasheet_params.json) is UTF-8 (ensure_ascii=
 false) so a µ/Ω/° in a datasheet-derived note no longer crashes the writer on
 the Windows cp1252 default. A NEEDS_CLARIFICATION that can't be resolved no
 longer loops setup forever: once acknowledged (clarification_acknowledged) the
 entry counts as fresh, so the run proceeds on the analyst's stated assumption.

-------------------------------------------------------------------------------
 B.5  SPICE-MODEL LIFECYCLE AGENTS  (generate / update / chat-edit)
-------------------------------------------------------------------------------
 The decks are dynamically dependent on the schematic. Three agents keep them in
 sync; all are GUARDED to the sim layer (may write sim/decks/*.py + edit
 blocks.yaml/service.py; must NOT touch netlist/altium/gen/linter — the
 schematic is the INPUT, never edited here). Shared _DECK_CONTRACT in agent.py.

   GENERATE   when a block has NO model (has_model=false, model_status=none)
     POST /api/sim/generate-model -> start_sim_generate_model
     Reads netlist + datasheets, studies existing decks as templates, WRITES
     sim/decks/<block>.py + service dispatch + a blocks.yaml entry, stamps
     provenance, verifies the sim runs.  GUI: "Generate SPICE model" button.

   UPDATE     when a model is STALE (netlist changed under it; model_status=stale)
     POST /api/sim/update-model -> start_sim_update_model
     Diffs the current netlist vs the deck's assumptions, makes the minimal
     changes to re-sync, re-stamps provenance, verifies.  GUI: "Update to match
     schematic" button.  Staleness = live netlist fingerprint != stamped one
     (sim/deck_provenance.py).

   CHAT-EDIT
     POST /api/sim/chat-edit -> start_sim_chat_edit
     Applies a natural-language request to a block's pass criteria / boundary
     params / model / functions (smallest change; CLARIFY if ambiguous).

===============================================================================
 C.  DESIGN-REVIEW PIPELINE  (the Review tab — closed-loop AI review)
===============================================================================

 A third pipeline over the SAME inputs (netlist + built schematic + datasheets/
 requirements): an AI design review that not only FINDS issues but iterates to
 FIX them, then lets you accept or reject the result.

   rules.yaml ──► evaluate (structural + semantic + sim_review) ──► findings
        │                                                              │
        │            ┌───────────────────  CLOSED LOOP  ──────────────┘
        ▼            ▼   (review/closed_loop.py, "Design review" button)
   ┌──────────────────────────────────────────────────────────────────┐
   │  per round (max 10, stop on all-clear / plateau×2 / cancel):       │
   │   Plan → Apply → Sim → Missing → Lint fix → Build → Re-eval        │
   └──────────────────────────────────────────────────────────────────┘
        │                                                  │
        ▼                                                  ▼
   review/findings.json  (final findings, flapping tagged)   Diff & Accept
   gui/state/loops/<id>.json (audit)                         (snapshot vs current)

 RULE MODEL (review/rule_schema.py, persisted in review/rules.yaml):
   113 rules today, all enabled (59 ERROR / 38 WARNING / 16 INFO).
   Families: schematic ("general checks", 78) · design ("components", 7) · block
   ("blocks", per functional block, 28). The "simulation" family is RETIRED.
   Evaluation: structural (Python predicate over the netlist — e.g.
   pullup_pulldown ×14, connector_pin ×20, net_routing ×15, power_rail_membership
   ×8, decoupling_count ×6, no_connect ×5, value_in_range ×2, present ×4) ·
   semantic (claude -p judges cited sources + design) · sim_review (×11 — a
   structural kind that RUNS ngspice then has claude -p judge the result, shown
   with a "simulated" tag). A full PASS = structural + semantic + sim_review all
   PASS.
   THE JUDGE (the ≈28 semantic + 11 sim_review verdicts): a read-only claude -p
   call per rule (rule_eval.py::_eval_semantic_rule / _judge_sim_result, perm-mode
   `plan`). Model = the `review_judge` agent kind (DEFAULT OPUS — the FINDERS get
   the same tier as the apply agent that fixes what they find; configurable in
   Design Resources → Agent Models, env override TEST1_AGENT_MODEL, thinking budget
   from the agent's effort). Each judge is a FRESH stateless call; its context is
   purpose-built by _netlist_context_for: the rule's cited source quotes + the
   subject part + the FULL membership of every net the subject touches (so a pull-
   up via a series-R is visible) + the whole sheet's parts + a deterministic
   MPN→value decode + the POPULATED-part datasheet facts + design_intent.md (the
   cross-sheet gotchas — same doc the apply agent reads, so find- and fix-side
   reason from the same intent). Fail-safe: any error/timeout/unparseable verdict
   → PASS (never fabricates a finding). Rules are managed MANUALLY in the GUI (add / edit / delete). Doc-driven
   rule GENERATION still exists in the backend (review/rule_gen.py + the rule_gen
   agent + POST /api/review/rules/generate) — it reads design_requirements.md +
   datasheets and proposes a rule set — but it is NO LONGER SURFACED in the GUI
   (the frontend has no generate button); the manual add/edit/delete path is the
   exposed workflow. load_rules is LENIENT — one malformed rule is reported under
   `rejected`, it no longer 500s the whole Rules panel.

 SEVERITY SCOPE (POST /api/loop/start → start_loop):
   review_only=True   → evaluate + report findings, run NO fix rounds ("Run review").
   fix_warnings=False → the loop clears ERROR findings only (WARNING/INFO advisory).
   fix_warnings=True  → "Fix errors + warnings": ERROR **and** WARNING findings are
                        in scope — they drive apply passes, the all-clear predicate
                        counts them, AND the post-build lint_fix triggers on (and is
                        prompted to clear) cosmetic lint WARNINGs too, not just
                        ERRORs. (in_scope() is the one gate; INFO never blocks.)

 THE CLOSED LOOP (review/closed_loop.py::run_loop), each round:
   Plan      plan_actions() buckets the IN-SCOPE findings → actions by fix kind.
   Apply     a claude -p agent (OPUS) edits altium/build_*.py + netlist. It's
             handed a rich changelog item PER finding (title · subject · refs ·
             Observed · the rule's fix_hint), the live-from-registry gate briefing
             (_lint_expectations: every lint rule + the anti-short idioms), and
             reads design_intent.md first. So apply gets the finding's actual
             detail, not "address rule X".
   Sim       ngspice re-checks affected blocks (only if a finding needs it).
   Missing   a referenced component with no library symbol → missing_part.py
             (search → rank → identity-check → author/install .SchLib → place →
             build → sim-verify → topology-adapt → impasse). build_project also
             FAILS GRACEFULLY on a missing symbol (lists the MPNs; no crash).
   Lint fix  if the rebuild left in-scope cosmetic lint findings (ERRORs always;
             +WARNINGs under fix_warnings), a lint_fix agent clears them — scope
             threaded through so its prompt matches the loop — then rebuilds once
             more. (lint.json is read via _read_lint_failures into the {issues,
             counts,status} shape the fix prompt consumes.)
   Build     python -m test1.altium.build_project (subprocess).
   Re-eval   SCOPED to the impacted area: only rules whose subject is on a
             CHANGED sheet (or whose refdes changed, or can't be localized) are
             re-checked; the rest carry their prior verdict forward (≈5-6× fewer
             slow semantic/sim calls/round). Rules that flip verdict across
             rounds with no explaining edit are flagged as FLAPPING.

 DIFF & ACCEPT (review/diff.py + the Diff panes):
   Pre-loop SNAPSHOT (altium/out/render_snapshots/<id>/) vs current. Content-aware
   refdes-level diff (added/removed/changed with anchor boxes) — rebuild noise
   shows nothing. When a change is too substantial to box meaningfully
   (re-layout / many adds-removes / unanchorable), renderable=false and the UI
   shows the two schematics plainly + WHY, instead of forcing a misleading
   overlay. Accept = archive the snapshot (keep the design). Reject = revert,
   with a VERIFY-AFTER-REVERT roll-forward guard: if reverting produces a worse
   state (build breaks / lint ERRORs rise) it ROLLS FORWARD to the post-loop
   state rather than stranding you. /api/loop/{id}/clear drops the residue
   (changelog + snapshot) without touching the design.

 Key Review endpoints (gui/backend/app.py):
   GET  /api/review/rules          rules + stale_sources + `rejected` (lenient)
   GET  /api/review/rules/pdf      the rule set as a PDF
   GET  /api/findings              current findings.json (envelope)
   POST /api/findings/{id}/apply   queue a PDF-finding's fix (Voltai-import path)
   POST /api/loop/start            start the closed loop (409 if one running)
   GET  /api/loop/{id}             summary (rounds, findings, flapping)
   GET  /api/loop/{id}/stream      SSE: eval_progress/round/action/build/flapping
   POST /api/loop/{id}/cancel      cancel (checked between phases)
   GET  /api/loop/{id}/diff        per-sheet diff (+ renderable / reason)
   POST /api/loop/{id}/accept | reject | clear
   GET  /api/png_snapshot/{id}/{sheet}   pre-loop render for the BEFORE pane

 The Workflow panel (gui/frontend/.../WorkflowSection.tsx — formerly

===============================================================================
 E.  GLOSSARY OF KEY INVARIANTS
===============================================================================
 • netlist/*.yaml is the single source of truth for both pipelines.
 • Generation is deterministic + gated (validator HARD, linter quality). No AI.
 • Sim decks are BEHAVIORAL; values come from the netlist (design_extract), not
   hardcoded; off-sheet boundary only lives in blocks.yaml.
 • Datasheet extraction is a SETUP-only (design-input) step; interpret is
   read-only on the param cache.
 • The iterate loop (interpret) and the model lifecycle (generate/update) are
   HARD-bounded / provenance-stamped so they converge and stay in sync.
 • Lifecycle agents may edit ONLY the sim layer; the schematic is never edited
   by a sim agent.
 • Review = a THIRD pipeline (C): rules.yaml → eval → closed loop (Plan/Apply/
   Sim/Missing/Lint-fix/Build/Re-eval) → Diff & Accept. Rules are manual; the
   loader is lenient; re-eval is scoped to the impacted area; reject is guarded
   by a roll-forward; a missing library symbol fails the build gracefully.
